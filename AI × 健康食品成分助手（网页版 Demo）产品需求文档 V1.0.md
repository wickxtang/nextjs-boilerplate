### 1. 产品定位与核心逻辑

**一句话介绍**：
这是一款帮助用户**看懂配料表背后风险**，并通过**双链关系**管理个人饮食图谱的健康工具。

**底层逻辑整合**：
- **风险维度**：WHO/IARC 致癌物等级（1类/2A/2B/3类）标注。
- **营养维度**：果蔬天然营养价值与包装食品高糖/钠/脂预警。
- **关系维度**：以 **“配料成分”为节点**，打通用户所有记录的零食，建立可视化关联。

**核心用户场景**：
> 用户拍下零食包装 → AI 解析配料表 → 风险提示与营养解读 → 存入个人零食库 → 配料自动转化为可点击的双链节点 → 点击节点查看所有含该配料的过往零食。

### 2. 功能模块详细设计

#### 模块一：AI 配料表智能解析（首页/录入页）

**2.1 输入交互**
- **方式 A（主力）**：图片上传/拍照识别。调用 OCR 接口识别文字。
- **方式 B（辅助）**：文本输入框（支持手动粘贴配料表文字）。

**2.2 AI 处理逻辑（调用 LLM API）**
- **实体识别与标准化**：将“抗坏血酸”映射为“维生素C”，将“E330”映射为“柠檬酸”。
- **风险分类**：
    - **红标**：含 1 类致癌物（如：加工肉类、酒精、槟榔）。
    - **黄标**：含 2A/2B 类致癌物（如：红肉、高温油炸物丙烯酰胺、阿斯巴甜）。
    - **蓝标**：高警示成分（如：果葡糖浆、氢化植物油、亚硝酸钠）。
- **生成解读**：生成一段 50 字以内的通俗解释（例如：“含有阿斯巴甜（2B类致癌物），但符合国标添加量，介意代糖后味者慎选。”）。

**2.3 输出结果页 UI 结构**
- **食品名称**：OCR 识别结果（允许用户修正）。
- **分解配料表**：按含量降序排列的胶囊标签。
- **快速评分卡**：致癌风险等级 / 营养密度（仅果蔬）/ 钠糖脂肪预警。
- **操作按钮**：**「存入我的零食库」**（点击后触发双链关系建立）。

---

#### 模块二：个人双链零食库（核心创新）

**2.4 零食库首页视图**
- **模式 A - 时间线视图**：按食用日期倒序排列，展示零食封面缩略图、风险标签。
- **模式 B - 全域图谱视图（视觉亮点）**：
    - 采用 ECharts Graph 力导向图。
    - **中心节点**：最近记录的零食。
    - **周围节点**：该零食包含的配料。
    - **连线规则**：点击配料节点后，图谱中心切换为该配料，周围扩散出**库内所有含有该配料的零食**。

**2.5 配料双链详情页（`/ingredient/[id]`）**
这是双链体系的**落地枢纽页**，包含以下区块：

| 区块名称 | 功能描述 |
| :--- | :--- |
| **百科头部** | 名称、WHO 致癌等级（配等级说明链接）、每日安全摄入量（ADI）。 |
| **反向链接区** | **“你摄入过的含有此成分的零食”**。列出具体零食名称、食用时间、摄入次数统计。 |
| **智能关联区** | **“未链接的提及”**。若用户在其他文本记录中提过相关词（如“无糖”），系统提示是否关联该配料。 |
| **替换建议区** | AI 推荐库内**不含此配料**的同类替代零食。 |

#### 模块三：健康知识库（信任背书）

**2.7 标准化说明页**
- **IARC 致癌物分级详解**：用通俗图表解释 1/2A/2B/3 类含义，强调“剂量决定毒性”。
- **数据来源声明**：IARC Monographs、中国 GB 2760 食品添加剂标准、USDA 食物成分数据库。

**2.8 果蔬营养查询（辅助入口）**
- 搜索框输入水果/蔬菜名。
- 返回 100g 标准营养成分表 + 核心健康益处简述。

### 3. 开发路线图与优先级

| 阶段 | 核心任务 | 验收标准 |
| :--- | :--- | :--- |
| **P0 (第一周)** | **AI 解析闭环** | 上传配料表图片 → 返回结构化的风险解读文字。 |
| **P1 (第二周)** | **零食库基础 CRUD** | 能将解析结果存入列表，支持手动增删改查。 |
| **P2 (第三周)** | **双链功能落地** | 配料名称变为链接；配料详情页展示反向零食列表。 |
| **P3 (第四周)** | **图谱可视化与优化** | 首页展示简单的力导向图 Demo，UI 打磨上线。 |

### 4. 技术栈建议

- **前端框架**：React / Vue (推荐 Vite 构建)。
- **AI 接口**：
    - OCR 识别：可采用开源库 Tesseract.js 或调用第三方云服务（百度 OCR / 腾讯 OCR）。
    - 语义解析：调用 DeepSeek / OpenAI API，配合精心设计的 Prompt 进行配料提取与映射。
- **图谱渲染**：ECharts 或 AntV G6。
- **数据存储**：Demo 阶段使用浏览器 IndexedDB 或 LocalStorage 存储个人零食库数据。


| 角色 | 技术选型 | 核心原因 |
| :--- | :--- | :--- |
| **前端** | **React** + **Vite** | 生态最丰富，图谱库（ECharts/AntV G6）对 React 支持最好；Vite 配置简单，秒级热更新。 |
| **后端** | **Next.js** (App Router) | **关键优势**：你可以把 AI 接口调用写在 Next.js 的 **API Routes** 里。这样你的前端不需要直接暴露 DeepSeek/OpenAI 的 API Key，也不会有跨域问题。**一套代码仓库同时解决前后端部署**。 |
| **数据库** | **SQLite** | Demo 阶段数据量极小，SQLite 文件即数据库，无需安装服务。零食和配料的关系完全适用关系型查询。 |

### 5. 后续迭代预留接口

- **扫码功能**：直接扫描商品条码获取标准配料表（需对接 Open Food Facts 等公开库）。
- **社交分享**：生成“我的健康饮食图谱”长图海报。
- **健康报告**：周期性统计“本月高频摄入的 2B 类成分 Top 3”。
